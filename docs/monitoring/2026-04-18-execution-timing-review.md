# 2026-04-18 실행 타이밍 점검

## 질문

현재 자동매매에서 진짜 중요한 것은 아래 흐름이 시간 순서대로 정확히 맞물리느냐이다.

1. 신호 발생
2. 주문 제출
3. 실제 체결 확인
4. 시스템 포지션 상태 반영
5. 청산 주문 제출
6. 실제 청산 체결 확인
7. 시스템 flat 상태 반영

즉, 전략 신호 자체보다 **주문과 체결 상태가 시스템 내부 상태와 언제나 일치하느냐**가 더 중요한 문제다.

## 결론

짧게 말하면 현재 코드는 다음 상태다.

- **진입 경로는 비교적 강하게 방어되어 있다.**
- **청산 경로는 진입만큼 대칭적으로 강하지 않다.**
- 따라서 "타이밍에 맞게 동작하면 문제 없다"는 말은 맞지만,
  현재 구현이 **매수/진입에는 꽤 가깝고, 매도/청산에는 아직 완전히 도달하지 못한 상태**라고 보는 것이 정확하다.

## 현재 구현 요약

### 1. 신호 탐색

메인 루프는 `src/strategy/live_runner.rs`의 `run()` 에서 돈다.

- 포지션 보유 중:
  - `manage_position()` 을 호출
  - 이후 3초 대기
- 포지션 미보유 중:
  - `poll_and_enter()` 로 신호 탐색
  - 이후 5초 대기

즉, 신호 자체는 완전한 틱 기반이 아니라 **5초 폴링 기반**이다.

관련 위치:

- `run()` 시작: `src/strategy/live_runner.rs:1540`
- 포지션 관리 루프: `src/strategy/live_runner.rs:1771`
- 신호 탐색 루프: `src/strategy/live_runner.rs:1849`
- 5초 폴링 주석: `src/strategy/live_runner.rs:1859`

### 2. 진입 주문

신호가 발생하면 `poll_and_enter()` 가 `execute_entry()` 를 호출한다.

현행 정책은 즉시 시장가가 아니라 **패시브 지정가 진입**이다.

- 의도한 진입가를 ETF 호가단위로 반올림
- 주문 직전 매수가능수량 재조회
- 진입 전 보유수량 스냅샷 확보
- 지정가 주문 제출
- 최대 30초 동안 체결 확인

관련 위치:

- `poll_and_enter()`: `src/strategy/live_runner.rs:1990`
- `execute_entry()`: `src/strategy/live_runner.rs:2683`

### 3. 진입 체결 확인

진입 후에는 곧바로 체결된 것으로 처리하지 않는다.

체결 확인 순서는 아래와 같다.

1. WebSocket 체결통보 대기
2. REST 체결조회 폴백
3. 미체결이면 취소
4. 취소 직후 다시 체결조회
5. 그래도 `filled_qty == 0` 이면 잔고 재검증

관련 위치:

- 체결 대기 루프: `src/strategy/live_runner.rs:2950`
- WebSocket 체결통보 대기: `src/strategy/live_runner.rs:4206`
- REST 체결조회: `src/strategy/live_runner.rs:4252`

이 구조는 `주문 넣었으니 체결됐겠지` 식이 아니라, **실제 체결 여부를 별도 루프로 확인하는 구조**다.

### 4. cancel/fill race 방어

현재 진입 경로에서 가장 잘 보강된 부분은 이 부분이다.

취소 직전 또는 취소 도중 주문이 체결될 수 있기 때문에, 체결조회가 0이어도 실제 보유가 늘었는지 잔고를 다시 확인한다.

관련 위치:

- cancel/fill race 감지: `src/strategy/live_runner.rs:3076`

즉, 진입 경로는 최소한 아래 오판을 줄이도록 설계되어 있다.

- 주문은 사실 체결됐는데
- 시스템은 미체결 취소로 착각해서
- 포지션이 없는 것으로 처리하는 문제

### 5. 포지션 관리와 손절 반응

청산 판단은 `manage_position()` 이 담당한다.

다만 손절 반응성은 3초 루프만 의존하지 않고, 별도의 실시간 가격 리스너가 보완한다.

- WebSocket 가격 수신 시 `latest_price` 갱신
- 보유 포지션이 있으면 틱 단위로 `best_price`, `stop_loss` 갱신
- 손절 돌파를 감지하면 `sl_triggered_tick` 플래그 설정
- `manage_position()` 이 이를 존중해서 즉시 청산 경로로 진입

관련 위치:

- 가격 리스너: `src/strategy/live_runner.rs:1405`
- 현재가 조회: `src/strategy/live_runner.rs:1513`
- 포지션 관리: `src/strategy/live_runner.rs:2460`
- SL 틱 감지 사용: `src/strategy/live_runner.rs:2652`

즉, 손절만큼은 신호 탐색보다 더 빠르게 반응하도록 구성되어 있다.

### 6. 익절(TP) 경로

익절은 TP 지정가를 미리 걸고, `manage_position()` 에서 체결 여부를 확인하는 구조다.

- 가격이 TP 수준 도달
- `fetch_fill_detail()` 로 실제 체결 여부 확인
- 부분 체결이면 잔여분은 시장가로 청산 시도

관련 위치:

- TP 체결 확인: `src/strategy/live_runner.rs:2519`
- TP 부분 체결 처리: `src/strategy/live_runner.rs:2534`
- 체결 상세 조회: `src/strategy/live_runner.rs:4092`

이 부분도 진입보다는 단순하지만, 최소한 **체결 확인 후 상태를 정리하려는 구조**는 들어가 있다.

### 7. 시장가 청산 경로

손절, 시간청산, 강제청산은 `close_position_market()` 으로 들어간다.

이 경로는 아래 순서로 동작한다.

1. 내부 `current_position` 을 임시로 꺼냄
2. 기존 TP 주문이 있으면 먼저 취소 시도
3. 시장가 주문 제출
4. 체결가 조회 시도
5. 체결가 미확인 시 현재가로 fallback
6. 결과를 trade record 로 저장하고 flat 상태 반영

관련 위치:

- 시장가 청산: `src/strategy/live_runner.rs:3429`
- 체결가 조회 사용: `src/strategy/live_runner.rs:3531`
- 체결가 미확인 시 현재가 fallback: `src/strategy/live_runner.rs:3535`

## 좋은 점

### 1. 진입은 즉시 체결 가정을 하지 않는다

현재 코드는 신호가 생기더라도 곧바로 포지션을 열린 것으로 처리하지 않는다.
이 점은 매우 중요하다.

### 2. WebSocket만 믿지 않는다

체결 확인에서 WS가 안 오면 REST로 보완하고, 취소 후에도 다시 확인한다.

### 3. 진입 race condition 방어가 들어가 있다

`filled_qty == 0` 이어도 실제 잔고가 늘었는지 확인한다.
이건 과거 사고를 반영한 실질적 보강이다.

### 4. 손절 반응은 신호 탐색보다 빠르다

손절 경로는 3초 루프만 보는 것이 아니라 실시간 틱 리스너가 먼저 감지한다.

## 약한 점

### 1. 신호 탐색은 아직 5초 폴링 기반이다

신호가 생긴 순간 바로 주문하는 구조가 아니라, 다음 폴링 사이클에서 잡힐 수 있다.

즉:

- 진입 신호 포착 지연 가능
- 빠른 ETF 움직임에서 FVG 존 재진입 타이밍을 살짝 늦게 볼 가능성

### 2. 진입 정책 자체가 패시브 지정가다

현재 라이브 정책은 공격적 체결이 아니라 **패시브 지정가 + 30초 대기**다.

이 말은 다음을 의미한다.

- 신호 타이밍과 체결 타이밍이 본질적으로 다를 수 있음
- 빠른 장에서는 진입이 늦거나 미체결될 수 있음
- 사용자가 기대하는 "신호가 떴으니 바로 들어가야 한다"와 구조가 다름

여기서 `30초`는 단순 구현 디테일이 아니라 전략 적합성 문제다.

- ETF 장중 단기 신호에서 `30초`는 짧지 않다
- FVG 재진입 신호는 수 초에서 수십 초 안에 성격이 바뀔 수 있다
- 따라서 현재의 `30초 대기`는 "기회 포착"보다 "늦은 체결 또는 뒤늦은 미체결"을 만들 가능성이 크다

즉, 현행 `30초`는 보수적 안정장치가 아니라, 이 전략 문맥에서는 **과도하게 긴 진입 유예시간**으로 보는 것이 맞다

관련 참고:

- `src/strategy/parity/execution_policy.rs`
- 현재 라이브 정책 설명: `PassiveTopBottomTimeout30s`

### 3. 시장가 청산은 진입만큼 강하게 재검증하지 않는다

현재 가장 큰 비대칭은 이 부분이다.

진입은:

- WS 확인
- REST 확인
- 취소 후 재확인
- 잔고 재검증

까지 간다.

반면 시장가 청산은:

- 주문 제출
- 체결가 조회 시도
- 실패하면 현재가 fallback

으로 끝난다.

즉, 아래 항목이 진입만큼 강하지 않다.

- 실제 체결수량 재확인
- 청산 후 잔고가 0인지 재확인
- 부분 체결 후 잔여수량 보유 여부 재검증

### 4. 청산 체결가 fallback은 기록 신뢰도를 낮출 수 있다

시장가 청산에서 체결가를 못 찾으면 현재가를 사용한다.

이 경우 아래 문제가 생길 수 있다.

- 실제 손익과 기록 손익이 어긋날 수 있음
- 청산은 됐는데 체결가 기록만 틀릴 수 있음
- 더 나쁘면 체결 자체가 불완전한데 flat 처리까지 이어질 위험이 남음

## 현재 답변

### "타이밍에 맞게 동작해야 문제가 없는 거 아닐까?"

맞다. 정확히 그 문제다.

이 시스템에서 본질적인 안정성은 신호 정확도보다 아래가 더 중요하다.

- 주문 제출 시각
- 실제 체결 확인 시각
- 상태 반영 시각
- 청산 판단 시각
- 청산 체결 확인 시각

이 순서가 엇갈리면, 좋은 전략도 실전에서는 망가진다.

### "현재가 그렇게 되어 있나?"

정확한 답은 아래와 같다.

- **진입 쪽은 상당 부분 그렇게 되어 있다.**
- **청산 쪽은 아직 완전히 그렇게 되어 있지 않다.**

좀 더 직설적으로 쓰면:

- `매수/진입`: 꽤 잘 보강됨
- `매도/청산`: 아직 entry 수준의 대칭성을 확보하지 못함

## 우선 보완이 필요한 지점

### 1. 시장가 청산 후 잔고 재검증 추가

`close_position_market()` 후 아래 확인이 필요하다.

- 청산 주문 체결 수량
- 현재 보유 수량
- 보유가 0이 아니면 부분 체결 또는 체결 지연으로 간주

이 검증이 없으면 "청산 주문은 냈다"와 "실제로 포지션이 없어졌다"를 같은 것으로 볼 위험이 있다.

### 2. 청산도 entry 수준의 WS+REST+balance 확인 구조로 맞추기

진입 경로의 강한 방어를 청산 경로에도 대칭적으로 적용해야 한다.

권장 순서:

1. WS 체결통보 대기
2. REST 체결조회
3. 청산 후 balance 재검증
4. 잔여수량 존재 시 후속 청산 또는 manual_intervention

### 3. 신호 탐색을 5초 폴링에서 바 이벤트 중심으로 줄이기

현재 구조는 운영상 안전하지만, 실행 타이밍 면에서는 늦다.

최소한 아래 중 하나는 필요하다.

- 5분봉 확정 이벤트 기반 진입 판단
- FVG 재진입 구간만큼은 더 짧은 주기 또는 틱 기반 감시

### 4. 패시브 지정가 정책 재검토

현재 전략이 "타이밍이 핵심"이라면,
패시브 지정가가 과연 맞는지도 별도 문제다.

실전에서 더 중요한 것이 체결 확실성이라면 다음 대안이 있다.

- 시장성 지정가
- best ask/bid + 1 tick
- 짧은 timeout 후 즉시 취소

특히 아래를 명시적으로 재검토해야 한다.

- 현재 진입 대기 `30초`
- 권장 진입 대기 `2~3초`
- 최대 허용 범위도 `5초`를 넘기지 않는 방향

## 최종 판단

현재 구현은 "신호가 발생했으니 체결됐겠지" 수준은 아니다.
특히 진입 경로는 체결 확인과 race condition 방어가 꽤 잘 들어가 있다.

하지만 아직 아래 문장까지는 말하기 어렵다.

> 신호 발생, 주문, 매수/매도 체결, 포지션 반영, 청산 반영이 모두 같은 수준의 신뢰도로 시간 정합성을 보장한다.

현 시점의 더 정확한 표현은 아래와 같다.

> 진입 경로는 실전 운용을 많이 의식해 보강되어 있지만, 청산 경로는 아직 같은 수준으로 대칭 보강이 필요하다.

## 후속 작업 제안

우선순위는 아래 순서가 적절하다.

1. `close_position_market()` 에 청산 후 잔고 재검증 추가
2. 청산 부분 체결/체결 지연 시 manual_intervention 또는 재청산 정책 추가
3. 신호 탐색 타이밍을 5초 폴링보다 더 촘촘하게 개선
4. 패시브 지정가 정책을 시장성 지정가 정책과 비교 검토
