### [문제 #1] WebSocket 영구 종료 후 프로세스 좀비 상태 — 분봉 0건 (09:03 KST 발견)

- **정의**: KIS WebSocket이 04:50 KST에 "최대 재시도 횟수 초과 — 종료"(critical)로 끊긴 후 재연결 시도가 완전히 멈췄으나, 프로세스 자체는 죽지 않고 axum HTTP 서버와 phase timer만 살아남아 UI상 "OR 수집 중"으로 정상 표시되고 있다. 실제로는 분봉 데이터가 한 건도 들어오지 않아 OR이 산출되지 못하고, 결국 FVG 탐색·진입·체결이 모두 불가능한 상태.

- **내용**:
  - 마지막 `minute_ohlcv` 레코드: **2026-04-14 13:06** (약 20시간 전)
  - 오늘 09:00 ~ 09:03 사이 `interval_min=1` 신규 분봉: **0건** (122630, 114800 모두)
  - `netstat -ano` 결과 PID 26948의 외부 연결: **DB(5433)·자기 자신(3000)뿐, KIS WS 서버 연결 없음**
  - phase_change 이벤트(id=834,835)는 09:00:00 정각에 정확히 발생 — phase timer 정상
  - 04:50 critical 이후 04:50~09:03 동안 신규 ws_reconnect/api_error 이벤트 0건 = WS 워커가 완전히 포기

- **원인**:
  1. **WS 워커 종료 시 프로세스 전체 종료가 트리거되지 않음** — main 루프와 WS 루프가 별도 task로 분리되어 있어 WS task가 panic/종료해도 axum/timer는 계속 동작.
  2. **재시작 자동화 부재** — critical 종료 시 supervisor/restart 메커니즘 없음.
  3. **04:25 기동 시점에 이미 KIS WS 측 거부 상태**가 누적되어 있었을 가능성 (전일 19:50 critical → 04:25 기동 → 즉시 재시도 10회 실패).
  4. 어제 19:50 사고와 동일 패턴 ([2026-04-14.md](./2026-04-14.md))이 재발했으며, 핫픽스가 "재발 방지"가 아니라 "재시작 시 정리"에만 집중되어 있었음.

- **근거**:
  - event_log id=832 (04:50:52 critical "최대 재시도 횟수 초과 — 종료")
  - event_log id=823~831 (04:27~04:48, ws_reconnect 시도 1~10/10 모두 실패)
  - event_log id=833 이후 09:00까지 4시간 동안 신규 시스템 이벤트 0건
  - 직접 DB 조회: `SELECT MAX(datetime) FROM minute_ohlcv WHERE interval_min=1` → 2026-04-14 13:06
  - netstat: KIS WS 엔드포인트(`ops.koreainvestment.com:31000`) ESTABLISHED 0건

- **즉시 조치 (장중)**:
  1. **kis_agent 프로세스 재시작** — 09:15 OR 1차 마감 전이라면 OR 일부 살릴 가능성 있음. 단 09:00~ 분봉을 KIS 분봉 조회 API로 backfill 필요.
  2. 사용자 확인 후 graceful 종료 (`Ctrl+C` 또는 일반 `taskkill`) → 새로 빌드된 release 바이너리로 재기동.

- **조치 결과 (09:07~09:09)**:
  1. `cmd.exe /c "taskkill /PID 26948"` (graceful 시도) → "이 프로세스는 /F 옵션을 사용하여 강제로 종료해야 합니다" — 콘솔 앱이라 graceful termination 불가 (ConsoleCtrlEvent 외부 전송 불가).
  2. 활성 포지션 0·미체결 주문 0 안전 확인 후 `taskkill /F /PID 26948` 강제 종료.

  4. KIS WS 연결 확인: `netstat -ano` 결과 `210.107.75.39:31000` ESTABLISHED, REST `210.107.75.32:29443` ESTABLISHED.
  5. OR 네이버 백필 자동 산출: 122630=100650/100100, 114800=1452/1448, or_source="naver", or_refresh_warnings=[].
  6. 분봉 7건/종목 적재 (09:00~09:07 KST). 09:00~09:05의 분봉은 새 프로세스 시작 직후 네이버에서 backfill됨.
  7. **결과**: OR 정상 산출, state="신호 탐색"으로 정상 전환 — FVG 탐색 시작 가능.

- **장 마감 후 수정 (코드 변경 후보)**:
  1. **WS 워커 critical 종료 시 프로세스 전체 종료**: `tokio::spawn`된 WS task가 critical 발생 시 `std::process::exit(1)` 또는 main에 종료 신호 전파. 외부 supervisor(systemd / Windows Service / nssm)가 자동 재기동하도록.
  2. **WS 헬스 워치독**: minute_ohlcv 마지막 갱신 시각이 N분 이상 정체되면 자동 재연결 트리거 (장중에만).
  3. **`/api/v1/strategy/status`에 WS 연결 상태 + 최신 tick 시각 노출** — UI에서 phase가 아니라 실제 데이터 흐름이 보이도록.
  4. **재시도 백오프 무한 모드**: 장중에는 max_retries 무제한, 60초 백오프로 계속 시도.
  5. **WS critical 시 알림** (Slack/이메일/Windows toast) — "조용히 죽는" 시나리오 차단.

- **대안**:
  - 즉시 재시작 대신 별도 백업 프로세스 기동 후 점진 전환 (현재는 단일 프로세스 구조라 미적용)

### [문제 #2] Yahoo OR 교체 영구 실패 가능성 — 장중 Yahoo 5분봉 게시 지연 (09:13 KST 발견)

### [문제 #4] 매수 미체결 1차 cancel 3회 연속 실패 + 에러 메시지 라벨 오류 (10:22 KST 발견)

  2. **1차 cancel 3회 실패 원인 후보**:
     - KIS API 일시 부하 (모의투자 환경 특유)

  1. **에러 메시지 라벨 수정**: cancel 함수에서 주문 종류(매수/매도/TP)를 인자로 받아 정확한 메시지 출력. 모니터링 가시성 + 디버깅 효율 개선.

  4. **첫 cancel 실패 시 sleep 후 재시도 vs 즉시 fallback** 전략 비교: 현재 retry → fallback 흐름이 효율적인지 검증.
  5. KIS 모의투자 cancel API 응답 코드별 분류 — 어떤 응답에서 실패했는지 raw 응답 로깅 추가.

- **대안 (당일 운영)**:
  - 핫픽스가 안전망으로 동작하므로 즉시 조치 불필요
  - 빈도 모니터링 (3회 이상 발생 시 P0 격상)
